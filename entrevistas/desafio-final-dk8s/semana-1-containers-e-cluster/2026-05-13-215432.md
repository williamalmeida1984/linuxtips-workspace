Nos bastidores, como os serviços `api-transacoes`, `api-contas` e `auditoria` interagem pra que essa operação seja concluída e registrada com sucesso?

### Resposta do aluno

Boa pergunta! Aqui está a resposta:

---

Quando o usuário faz a transferência de R$ 50, o fluxo acontece assim:

**1. api-transacoes recebe a requisição**
O cliente chama `POST /transferencias` na `api-transacoes` com `origem_id`, `destino_id` e `valor`.

**2. api-transacoes chama api-contas**
A `api-transacoes` faz duas chamadas HTTP para a `api-contas`:
- Debita R$ 50 da conta de origem
- Credita R$ 50 na conta de destino

A `api-contas` valida se a conta existe e se tem saldo suficiente. Se não tiver saldo, retorna erro e a transferência não é concluída.

**3. api-transacoes publica o evento na auditoria**
Após o débito e crédito serem confirmados, a `api-transacoes` faz um `POST /eventos` na `api-auditoria` com os detalhes da transação — IDs das contas, valor, timestamp e status.

**4. auditoria persiste o evento**
A `auditoria` grava o evento em um arquivo JSONL em `/data/eventos-YYYY-MM-DD.jsonl`. No nosso cluster esse `/data` está montado num PV NFS compartilhado entre as 3 réplicas.

**5. api-transacoes retorna o resultado**
Só depois de tudo isso a `api-transacoes` retorna `{"status":"concluida"}` para o cliente.

---

Em resumo: `api-transacoes` é a orquestradora — ela coordena `api-contas` para mexer nos saldos e `auditoria` para registrar o evento. As três se comunicam via HTTP interno usando os FQDNs dos Services do Kubernetes.

Qual é a principal vantagem de segurança que a gente ganha usando Distroless e que justifica essa complexidade extra no build?

### Resposta do aluno

Boa pergunta! Aqui está a resposta:

---

A principal vantagem é a **redução drástica da superfície de ataque**.

Quando usamos `python:slim` no runtime, a imagem ainda vem com shell (`/bin/sh`, `/bin/bash`), package manager (`apt`, `pip`), utilitários de sistema (`curl`, `wget`, `ls`, `cat`) e outras ferramentas que não são necessárias para a aplicação rodar.

Isso é um problema porque se um atacante explorar uma vulnerabilidade na aplicação — por exemplo, uma injeção de código — ele consegue:
- Abrir um shell interativo
- Instalar ferramentas maliciosas com o `pip` ou `apt`
- Usar `curl` para baixar payloads externos
- Explorar os binários do sistema para escalar privilégios

Com **Distroless**, nada disso existe. A imagem tem apenas o Python runtime e as libs da aplicação. Sem shell, sem package manager, sem binários extras. Então mesmo que o atacante explore uma falha na aplicação, ele fica "preso" — não tem ferramentas para se mover lateralmente.

Além disso, menos pacotes significa menos CVEs. No nosso scan com Trivy, o `python:slim` tinha 27 vulnerabilidades HIGH/CRITICAL. Com Distroless + atualização das dependências, chegamos a zero.

E combinando com o usuário **nonroot (UID 65532)**, mesmo que algo seja comprometido, o processo não tem permissão de root no host.

---

Em resumo: Distroless não impede que a aplicação seja explorada, mas **limita drasticamente o que o atacante consegue fazer depois** — que é exatamente o princípio de defesa em profundidade.
